# gh-issue-treefier

## プロジェクトの概要

GitHub issue の依存関係（blocking/blocked by）を DAG として可視化・編集する GitHub CLI の拡張機能です。web ベースの UI を提供し、issue 間の関係を直感的に操作できます。

spec-workflow を使用した仕様駆動開発を行います。ステアリングドキュメント (.spec-workflow/steering/) や仕様・設計・タスク (.spec-workflow/specs/) をこのリポジトリで管理しています。具体的な仕様や技術的な構成・設計方針については、これらのドキュメントを参照してください。

## 開発

### CLI

GitHub CLI の拡張機能は Go 言語で実装されています。リポジトリのルートに `go.mod` と `go.sum` が存在します。
適宜 `go mod tidy` と `go fmt` を実行し、コードの整合性とフォーマットを保ってください。

コードが変更された場合は、それに対応するテストコードがあるか確認し、必要に応じて追加・修正を行ってください。テストの修正の有無にかかわらず、変更後は必ずテストを実行して動作確認を行ってください。

### Web

Web UI は TypeScript と React で実装されています。`web/` ディレクトリにソースコードが存在します。
適宜 `npm run lint` と `npm run format` を実行し、コードの整合性とフォーマットを保ってください。
コンポーネントに対しては storybook と storybook のインタラクションテストを使用してテストを行うようにしてください。API 通信部分については MSW を使用して API モックを作成してください。関数やユーティリティについては Vitest を使用してユニットテストを作成してください。
コードが変更された場合は、それに対応するテストコードがあるか確認し、必要に応じて追加・修正を行ってください。テストの修正の有無にかかわらず、変更後は必ずテストを実行して動作確認を行ってください。

## 開発フロー

spec-workflow を使用した仕様駆動開発を行います。新しい機能追加や大きな変更を行う場合は、まずステアリングドキュメント (.spec-workflow/steering/) に概要と目的を記載したドキュメントを作成し、チームメンバーと共有してください。その後、具体的な仕様・設計・タスク (.spec-workflow/specs/) を作成し、実装に取り掛かってください。各タスクの完了後は、対応する仕様ドキュメントを更新し、変更内容を反映させてください。
